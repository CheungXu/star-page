# 技能（Skill）发现与注入

把"领域技能/最佳实践"按需喂给 LLM，提升特定场景产出质量的通用做法。以"网页制作技能"为例，但适用于任何"按输入意图选用一段专项指令/知识再生成"的场景。

## 核心结构：技能目录 + 选择器 + 注入

1. 技能目录（数据源）：一个技能一个文件夹，`SKILL.md = frontmatter(元数据) + 正文(指令)`。元数据含 `key/name/description`（描述是路由判断的主要依据），可选 `triggers`（关键词兜底）、`enabled`。进程级缓存，约定优于配置（目录即数据源，复刻"模型目录 Registry"模式）。
2. 选择器（如何选）：抽象成接口，便于演进。三种范式由轻到重：
   - 关键词/规则匹配：零额外调用、最快，但不够智能。
   - 轻量 LLM 分类路由（推荐起步）：把技能 `name+description` 清单交给模型，让它只返回一个 `key` 或 `NONE`。准、实现简单。
   - Agentic 工具发现：system 只放技能清单，给模型 `list_skills`/`read_skill` 工具，模型按需读取全文再生成。最智能、最省 token（技能多时），但需要 LLM 客户端支持 tool calling + 生成端多轮循环。
3. 注入（怎么用）：选中技能正文包装成一条 system 消息，叠加在通用 system 之上；首轮与多轮（续写）都注入。

## 工程经验

- 默认开启 + 全自动路由：前端无选手动入口，后端在各 task 执行时按生成模型自动分类；用户无感享受能力。
- 路由必须有兜底且不可阻断主流程：给较短超时（路由只是"选一个词"），失败/超时回退关键词、再回退不注入。模型路由用截断后的"意图 prompt"（而非含附件的超长上下文），省 token、提速。
- 选择时机选 task 级、按生成模型路由：多模型并行时，各 task 用**生成 HTML 的同一 model_key** 做轻量分类，不同模型可匹配不同技能；路由 token 与生成 token 合并计入成本。
- 持久化 + 多轮延用：把选定的 `skill_key` 落到批次/节点/任务；续写沿父链路延用首轮技能，保证整条对话风格一致并省调用。
- 渐进式披露省成本：技能多且大时，不要把全部正文塞进每次请求；用"先给清单、再按需取全文"（Agentic）或"先分类命中、只注入命中项"（路由）控制上下文规模。轻量技能（1-3KB）可直接全量注入。
- 选择器做成接口，平滑演进：B（LLM 分类）与 C（Agentic）共享技能目录与注入逻辑，只替换"如何选"，演进不动注入与持久化。
- 部署陷阱：技能目录若在仓库根而服务构建上下文在子目录，容器里会"找不到技能"而静默退化为通用生成。需把技能目录 COPY 进镜像或挂载，并用环境变量指向其路径；目录定位逻辑应支持绝对路径与相对路径逐级向上查找。
- 复用外部开源技能要先"裁剪到自身形态"：上游 skill 常假设不同的运行环境（如可读写文件、React/Next、需 Figma/部署/测试执行）。改编时只取与自身产物形态匹配的内容（本项目=单文件自包含静态 HTML），剔除环境不匹配项，并显式记录"采用/未采用及原因"。把原始件放在被版本库忽略的参考目录（如 `ori-page-skills/`），成品才进 `page-skills/`，避免大体积与嵌套 `.git` 污染仓库。
- 通用底线放"基础提示"、差异化放"技能"：路由一次只命中一个技能，若把"可访问性/反 AI 套路感"等对所有产出都该成立的底线做成可路由技能，会因未命中而漏掉。正确做法是把这类通用基线并入基础系统提示（每次生成都注入、人人受益），技能只承载"某类页面/某种强化"的差异化内容，二者形成"底线 + 专项"的层叠。
