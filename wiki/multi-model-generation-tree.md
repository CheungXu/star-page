# 多模型并行生成与生成树（会话=树，Node=Page）

面向"一次请求并行多个模型、可对比、可多轮迭代、可分支"的生成式产品，记录一套兼顾当前需求与未来扩展的数据/交互结构。

## 三层结构

- 会话（conversation）= 一棵生成树：一次"创作"在多轮迭代中始终是一棵树。
- 批次（batch）= 一轮生成：持有该轮共享数据（prompt、文件、抽取文本、选中模型）与树关系（基点 `base_page_id`、类型 `kind`、合并来源 `source_page_ids`）。
- 节点（node）= 某个模型的一个结果：直接映射为可独立分享的 `Page`（复用既有 `/p/{page_id}` 访问网关），各自有产物版本与 token 记录。

一轮内每个模型一个节点（兄弟节点共享同一个 `parent`）。"续写"永远单父，所以会话内是一棵严格的树，便于索引与导航。

## 为什么 Node = Page

- 每个模型结果天生需要独立可分享链接（用户可能更想要其中某个、或在不同阶段/不同受众用不同结果）。
- 复用既有页面访问网关与"固定视口缩放"预览，serving 不改。
- 老数据等价于"单节点单批次会话"，迁移成本低。

对比："一个 Page 多个候选版本"在分支/多轮场景下，版本号线性语义与分支会冲突，serving/权限改动更大，不推荐。

## 合并：另起新会话，而不是把会话变成 DAG

多个节点融合时，如果留在同一会话里就会形成多父 DAG，会话越大越难索引、用户越容易混乱。更好的做法：

- 合并 = 新建一个会话，其根批次 `kind=merge_seed` + `source_page_ids=[跨会话节点...]` 记录血缘。
- 每个会话始终是一棵干净的树；会话之间用 merge 边形成更高层的松散关系，复杂度被限制在"会话内是树"。

## 并行请求与进度展示

- 每个模型 run 是一个独立任务，有自己的事件流；前端为每个 run 开一路 SSE（`EventSource`），按 `task_id` 把事件路由到对应 run 的状态。
- 左侧用"本轮模型" tab 切换查看各模型的思考与创建节点，避免 N 份长列表堆叠；右侧并排对比各模型结果。
- 批次状态由兄弟 run 聚合：未完成=running，全成功=succeeded，部分成功=partial，全取消=cancelled，否则 failed。每个 run 结束时幂等重算，轻微竞态可接受。

## 多轮续写的上下文策略（带哪些历史）

续写本质是"在上一版产物上改"。对"生成完整 HTML"这类产品，产物本身是完整快照，所以上下文按"指令全带、答案只带最近一版"组织最划算：

- system：通用 HTML 规则（不变）。
- user：历次"指令"清单（首轮原始需求 + 每一轮续写指令，按时间正序）。指令短、便宜，完整保留用户意图，避免"只看 HTML 推不出的约束"丢失。
- assistant：只放最近一版完整 HTML。每版 HTML 都是完整文档，已包含此前全部改动，更早版本是冗余的，多带只会撑爆上下文。
- user：本轮新指令（追加一句"未提到的部分尽量保持不变"，抑制整页重写时的无意漂移）。

并行续写时，每个模型沿"自己那条链"向上收集指令与取自己上一版 HTML（parent_page_id 天然区分链路），互不串味。

"回到更早版本"不靠上下文记忆，而靠生成树——从任意旧节点点「以此结果继续」分支即可，比让模型记历史更可靠。

注意：单次输出受 `max_tokens` 限制（如 8192），复杂页面可能被截断；续写时会把"上一版（可能被截断的）HTML"作为上下文，模型需重建完整页面。页面偏复杂时应调高对应模型 `params.max_tokens`。

## 扩展性与注意点

- N 路 `EventSource` 受 HTTP/1.1 单源约 6 连接限制：默认 2 个无碍；模型变多后升级为"批次多路复用 SSE"（一条连接合并多 run，事件带 `run_id`）或轮询。
- 模型可选集合来自"模型目录 + 多选"，加模型只改配置；并行数量从 2 扩到更多无需改结构。
- 续写/分支只需新建一个 `base_page_id` 指向某节点的批次；合并只需新建带 `source_page_ids` 的新会话根。
