# 生成页面预览设计原则

## 背景

在“自然语言生成 HTML 页面”这类产品中，用户需要在主站内预览生成结果。一个常见误区是把生成页面直接放进右侧窄容器 iframe 中渲染。

如果生成页面本身有响应式布局，窄 iframe 会触发移动端断点，导致预览效果与用户单独打开页面时完全不同。例如单独打开时是桌面端宫格布局，但预览中变成单列竖排。

## 推荐方案

预览应尽量模拟用户最终打开页面的真实视口，而不是使用预览容器的实际宽度。

推荐做法：

```text
生成页面
  -> 在 iframe 内使用固定桌面视口宽度渲染，例如 1200px
  -> 根据预览容器宽度计算缩放比例
  -> 根据预览容器高度反推 iframe 视口高度
  -> 对 iframe 整体 scale 缩放展示，并尽量铺满预览区域
```

这样可以让用户看到更接近“单独打开页面”的整体布局，细节可以缩小，但结构不应被窄容器改写。

## 实现要点

- iframe 内部宽度固定，例如 `1200px`。
- 外层容器根据可用宽度计算 `scale = containerWidth / viewportWidth`。
- iframe 高度不要直接使用生成页面整页 `scrollHeight`。
- 推荐按预览区域可用高度反算：`iframeHeight = previewAvailableHeight / scale`，并设置一个合理的最小桌面视口高度，例如 `900px`。
- 使用 `transform: scale(...)`，并设置 `transform-origin: top left`。
- 外层 shell 使用缩放后的宽高占位，避免布局塌陷。
- 外层预览区域可使用 `overflow: hidden`，优先保证首屏视觉铺满；如后续需要浏览长页面，应单独设计预览内滚动或“打开页面”入口。

## 避免 `100vh` 膨胀

一个真实踩坑是：如果 iframe 高度设置为生成页面整页 `scrollHeight`，生成页里的 `100vh` 会以这个超高 iframe 为基准计算。结果首屏 hero 的内容被垂直居中到很下面，预览里只看到大块背景色，看起来像页面坏了。

因此，预览高度应模拟“浏览器视口高度”，而不是“文档总高度”。主站右侧预览卡片只需要展示最终页面的桌面首屏视觉；完整长页浏览交给“打开页面”。

## 与直接 iframe 的区别

直接 iframe：

```text
预览区宽 500px
  -> iframe 宽 500px
  -> 生成页面触发移动端布局
  -> 用户看到的不是最终桌面效果
```

固定视口缩放：

```text
预览区宽 500px
  -> iframe 仍按 1200px 渲染
  -> scale 到 500px 显示
  -> 桌面布局保持，整体缩小
```

## 适用场景

- AI 生成落地页、产品页、报告页、展示页。
- 用户关心整体视觉和排版结构。
- 生成页面会通过独立链接被打开。

## 不适用场景

- 需要测试移动端响应式效果时，应提供单独的移动端预览模式。
- 需要逐像素编辑页面时，应提供更专业的画布和缩放控制。
- 页面包含复杂交互或跨域资源时，需要额外处理 iframe 安全和通信。

## 预览容器的视觉外观：做成"真实浏览器视窗"

缩放方案解决的是"渲染什么尺寸"；容器外观决定"看起来像不像最终产物"。

常见反模式：深色预览内容外围包一圈较宽的白色 padding，放在浅灰背景上，看起来像"挂在墙上的画"，笨重且缺乏沉浸感。

推荐做法——把预览容器做成一个**极简浏览器视窗**：

- 外层只用 1px 细边框（如 `#E5E7EB`）+ 一层细腻弥散阴影（Drop Shadow），去掉厚白边。
- 顶部加一条极简浏览器控制栏：左侧红 / 黄 / 绿三个交通灯小圆点（`#ff5f57 / #febc2e / #28c840`），右侧一个弱化的浅灰胶囊地址条显示当前预览 URL。控制栏放在缩放舞台**之外**（不占缩放计算的高度），这样 `scale` 反算逻辑无需改动。
- 缩放舞台（承载 scaled iframe 的区域）用干净的浅灰底，避免棋盘格等花纹增加视觉杂乱。
- scaled 页面本身收一个小圆角（如 10px）+ `overflow: hidden` + 细边框，像一张浮在舞台中的页面。

布局上让 `预览窗口` 为 flex column：固定高度的控制栏 + `flex:1` 的舞台；用舞台的 `clientWidth/Height` 做 scale 计算，控制栏不参与，避免改动既有缩放逻辑。

效果：预览区从"挂画"升级为"真实 Web 视窗"，高级感和沉浸感明显提升。

## 经验总结

主站内预览的目标不是“把 HTML 塞进一个区域”，而是“让用户尽早判断最终链接打开后的视觉效果”。对于生成式页面产品，预览应该优先还原最终消费场景；容器外观尽量贴近真实浏览器视窗（细边框 + 弥散阴影 + 极简控制栏），而不是厚白边的"画框"。
