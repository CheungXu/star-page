# AI 生成过程可视化与状态反馈 UX 原则

面向"AI 生成 / 长任务 / 流式处理"类产品（生成网页、报告、代码、图片等）的过程可视化与状态反馈设计原则。核心目标：在等待期把"系统在干什么、进行到哪一步、还要不要操作"讲清楚，同时保持视觉克制，让焦点落在最终交付物上。可跨项目复用。

## 1. 步骤条（Stepper）要有"流程感"而非"卡片堆叠"

独立卡片简单堆叠缺乏递进感。在节点圆圈之间补一条垂直连线，并按状态区分：

- **已完成**：成功色实线。
- **进行中**：品牌色"流光虚线"——用 `repeating` 渐变 + `background-position` 动画做向下行进的视觉，强表达"正在进行"。
- **未开始**：浅色实线（或浅描边）。

实现要点：连线用进度项的 `::after` 绝对定位在图标列水平中心，`top` 落在图标下沿、`bottom` 取负值跨过卡片间距连到下一节点（如 `top:40px; bottom:-16px` 配合 16px 列表间距）。图标 `position:relative; z-index:1` 压在连线之上。

```css
.step:not(.is-last)::after {
  content: ""; position: absolute; left: 29px; top: 40px; bottom: -16px;
  width: 2px; transform: translateX(-50%); background: var(--border);
}
.step.done:not(.is-last)::after { background: rgba(22,163,74,.4); }
.step.running:not(.is-last)::after {
  background-image: linear-gradient(180deg,var(--primary) 0 45%,transparent 45% 100%);
  background-size: 100% 12px; background-repeat: repeat-y;
  animation: flow .9s linear infinite;
}
@keyframes flow { to { background-position: 0 12px; } }
```

## 2. 状态色要"克制"——只强调"进行中 / 失败"，不强调"已完成"

常见反模式：所有节点完成后整片卡片铺满浅绿背景，视觉比重过大，反而**抢走核心交付物（如右侧预览）的焦点**。

原则：成功状态已由"绿色 Check 图标 + 绿色连线"表达，**不需要再用整块卡片背景强调**。

- 已完成节点 → 回归白底 / 透明 + 默认浅边框。
- 进行中节点 → 可保留浅品牌色高亮（值得强调）。
- 失败节点 → 保留红色背景（错误需要被看见）。

让用户视觉焦点自然过渡到真正关心的产出区。

## 3. "进行中"用动效，不要用静态点

- 进行中节点圆圈内放转圈 spinner（小尺寸 border 环旋转），比静态圆点更能表达"正在处理"，可再加一圈柔光 `box-shadow`。
- 进度区顶部 / 预览区的状态指示点，生成中用**蓝色"呼吸点"**（扩散光环动画），而不是绿点——绿点在 UI 语境里默认代表"在线 / 成功 / 完成"，生成途中用绿点会误导。完成后再转绿，失败转红，并配状态文案。

## 4. 处理中的主操作按钮：明确"不可点"或"可打断"

高亮蓝实心按钮在处理中仍是实心，会被误认为可点击。两种正确处理：

- **不可打断**：降级为加载态——浅色背景 + 主题色文字 + 转圈 spinner + `cursor: progress`，明确"系统忙"。
- **可随时打断**：文案改"停止生成" + 方形 Stop 图标 + 中性 / 警示色（需后端支持取消时才用）。

## 5. 全局只保留一个主 CTA（Primary Button）

同屏出现两个高亮主按钮会让操作焦点涣散。判断"当前状态下最核心的下一步动作"是哪一个，只让它用主 CTA 实心样式，其余降级为次级（浅色底 + 主题色文字 / 描边）。

例：页面生成完成后，"打开页面"是最终出口（主 CTA），底部输入框的"迭代修改"入口应降级为次级。

## 6. 按钮文案要贴合当前语义

页面已生成后，底部输入的动作其实是"迭代修改"，再叫"创建"词不达意。改为"发送 / 修改"或纯纸飞机(Send)图标更准确。文案要随状态切换（首屏"创建"、工作区"发送"、处理中"生成中"）。

## 7. 等待区用骨架屏（Skeleton）而非空图标

预览 / 结果区在生成期间不要只放一个小图标 + 长文案（空旷且重心偏上）。用"目标产物的线框骨架屏 + 流光 shimmer 动画"占位，既有信息量又缓解等待焦虑、强化科技感。

```css
.sk { background: linear-gradient(90deg,#eef1f6 25%,#e1e6ef 37%,#eef1f6 63%);
      background-size: 400% 100%; animation: shimmer 1.4s ease infinite; }
@keyframes shimmer { 0%{background-position:100% 0} 100%{background-position:0 0} }
```

## 8. 辅助数据（如 Token 用量）做成"科技感微徽"

把 `· 输出 N tokens` 这类辅助信息从"像无序列表默认小圆点"升级：前缀换成闪电 ⚡ / 数据库等线性微图标，文字用更弱的次级灰（如 #9CA3AF / muted），并与上方描述文字**完美左对齐**，作为自然融入卡片的补充信息，而非悬浮 Tag。

## 9. 选中态用"字重对比"强化定位

侧边栏 / 列表当前项除了背景色 + 左侧竖条，还应让**标题字重**与未选项拉开差距（未选 500、选中 700）。字体粗细对比是"你在这里"最直接的感知信号。

## 10. 滚动条要"隐形且优雅"

精致 UI 中不要出现浏览器默认粗滚动条。自定义 `::-webkit-scrollbar`：宽 4–6px、`border-radius: 999px` 圆角、滑块浅灰半透明（静止态 alpha ~0.16 近隐形），hover 才加深；`scrollbar-width: thin` 兼顾 Firefox。

## 11. 所有装饰性动画都要做无障碍降级

`stepper-flow` 流光、spinner、呼吸点、skeleton shimmer 等装饰动画统一纳入 `@media (prefers-reduced-motion: reduce) { animation: none }`。

## 适用 / 不适用

- 适用：任何"提交后需要等待、分阶段完成"的产品——AI 生成、批处理、流水线、长查询。
- 不适用：纯即时响应、无中间过程的交互，过度的过程可视化反而是噪音。

## 经验总结

过程可视化的目标不是"把所有状态都堆给用户看"，而是**克制地讲清进度、明确唯一焦点、用动效区分"进行中"**。色彩面积、主按钮数量、滚动条粗细这些"减法"，往往比加功能更能把产品从"好用的工具"推向"高级的产品"。
