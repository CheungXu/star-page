# 积分计费 + 复式记账 + 匿名风控通用经验

面向「AIGC / 按次消耗」类产品的计费体系通用做法，可跨项目复用。星页落地见 `doc/20260614/`、`doc/20260615/`。

## 一、积分模型：整数存储 + 双桶

- **单位**：积分整数存储，`1 元 = 100 积分`。全程整数避免浮点误差，展示时再换算成元。
- **扣费**：`扣费积分 = max(ceil(原始成本(元) × 模型倍率 × 100), 1)`。
  - 向上取整 + 最小 1 积分兜底：避免极小成本取整为 0 被白嫖。
  - 倍率按模型可配、可 `<1`（打折）。原始成本沿用既有的 LLM 用量成本算法。
- **双桶**：钱包分 `充值积分(paid)` 与 `赠送积分(gift)` 两个余额。消费时**先扣 gift、后扣 paid**，二者会计处理不同（见下）。
- **快照**：每条消费流水快照 `原始成本 / 倍率 / 确认收入`，配置变更不污染历史。

## 二、复式记账：积分是预收账款，收入在消费时确认

把「积分」视为**预收账款（递延收入）**，**充值时不确认收入，消费时才确认**。区分充值与赠送两条负债：

会计科目：资产 `1001 现金`、`1102 预付账款-云/LLM供应商`；负债 `2001 预收账款-充值积分`、`2002 赠送积分负债`；收入 `5001 服务收入`；费用 `6001 算力成本(COGS)`、`6002 基础设施成本`、`6601 推广赠送费用`。

业务事件 → 分录：
- 充值成功 X 元：借 `1001 现金` X / 贷 `2001 预收-充值` X。
- 赠送发放 面值 F：借 `6601 推广费` F / 贷 `2002 赠送负债` F（未用赠送 = 2002 余额）。
- 消费（充值部分 p、赠送部分 g 积分，原始成本 C 元）：
  - 借 `2001` p×0.01 / 贷 `5001` p×0.01；
  - 借 `2002` g×0.01 / 贷 `5001` g×0.01；
  - 借 `6001 COGS` C / 贷 `1102 预付账款` C（毛利 = 确认收入 − C）。
- 匿名试用消费：仅记 `6001 COGS` / `1102 预付`，无收入（计入试用获客成本）。

财务总览即由各科目借贷净额推导：充值现金=1001 借方累计、确认收入=5001 贷净、COGS=6001 借净、毛利=收入−COGS、赠送未用=2002 贷净、预收=2001 贷净。

### 关键：供应商是「预付费」还是「后付费」决定用资产还是负债

- **后付费/月结**（先用后还账单）：用 **负债** `应付账款-供应商`，消费时 贷应付，月底付款时 借应付/贷现金。
- **预付费**（先给云账户充值、调用即扣，国内云普遍如此）：用 **资产** `预付账款` 更准。现金在**充值那一刻**就花了（借预付/贷现金），消费是**冲减预付资产**（借成本/贷预付）。此时该资产净额＝云账户账面剩余，能直接和厂商真实余额对账。**误用「应付」会让资金状况完全反向**。
- 推论：若用 `payment_amount`（现金支付）当成本会严重做低——现金是提前一次性付的，账单期内几乎不再付现。成本应取 `pretax_amount`（折扣后应付＝资源消耗公允价值）。

### 财务总览分三段更清晰

- 付费业务（不含赠送）：真实付费现金业务的收入/成本/毛利。
- 赠送台账：赠送发放/未用负债/已核销收入/赠送+试用成本。
- 含赠送合计 + 营业利润（= 综合毛利 − 服务器等基础设施期间费用）。
- 收入与成本按消费流水 paid/gift 占比拆分（成本按占比归集，用尽=0 的匿名/未回收单列为试用获客成本）。

## 三、幂等是第一原则

- 所有引起余额变化的写入都带 `idempotency_key`：充值用 `recharge:{order_id}`、消费用 `consume:{version_id}`、赠送用 `gift:signup:{user_id}`。
- 复式凭证按 `(event_type, event_ref)` 唯一，重复过账自动跳过。
- 充值入账用 `UPDATE ... WHERE id=? AND status='pending'` 原子流转 + 唯一约束，防重复回调重复发积分。
- 过账前断言借贷相等（带浮点容差），不平衡直接抛错，避免脏账。
- 余额列加 `CHECK >= 0`；结算时 `扣减 = min(应扣, 余额)`，单次小额兜底不阻断已生成结果。

## 四、匿名 → 注册：用真实用户行建模匿名

- 把匿名访客建模为 `users` 中 `is_anonymous=true` 的真实行，所有 `conversations/pages` 外键无需改造即可挂载；注册时把匿名数据 `reassign` 到正式账号并标记 `merged_into_user_id`。
- `resolve_actor`：有登录态返回登录用户；否则校验签名 cookie 取匿名用户，无则懒签发。
- 读写分离两个解析器：写操作（生成）用会创建匿名身份的 `resolve_actor`；只读（历史/账户）用不创建、不写 cookie 的 `get_optional_actor`，避免「只浏览也建一堆匿名行」。

## 五、匿名防滥用：分层围栏，性价比优先

裸 cookie 有两类滥用：清 cookie/无痕刷新身份；篡改/伪造绕过计数。MVP 对策：

1. **签名 cookie（HMAC）**：服务端用密钥对匿名 id 签名，只认自家签发；想要新 id 必须走签发逻辑，便于限流。
2. **IP 天花板**：记录签发 IP，限制每 IP 每日签发匿名 id 数与免费生成次数；即使清 cookie 也封住批量薅。
3. **成本兜底**：匿名只能选次优模型、单次 ≤2 个模型；**后端强校验**（不只依赖前端置灰）。高价模型前端「可见但置灰」引导注册，而非隐藏。
4. 后续增强（非 MVP）：设备指纹、验证码、风控评分。

## 六、充值资金安全（第一天就硬）

- **价格服务端权威**：客户端只传套餐 key，金额/积分服务端算，杜绝「充 1 元得 1 万积分」。
- **mock 生产禁用**：自助 mock 支付仅 `APP_ENV != production` 开放。
- **真实回调**：先验签后解密，校验订单归属 + 金额一致 + `trade_state=SUCCESS` 才入账。
- 全程写积分流水 + 复式分录，支持后台对账发现差异。

## 六之二、微信 Native 支付接入要点（可复用）

- **验签选「微信支付公钥」模式**：免平台证书轮换（每 5 年换一次），初始化只需 `public_key + public_key_id`；但**商户私钥 `apiclient_key.pem` + 证书序列号永远必填**（用于给「发出的请求」签名，与验签模式无关）。注意证书 zip 里的 `apiclient_key.pem` 才是商户私钥，下载的 `pub_key.pem` 是微信支付公钥（验签用），别弄混。
- **库与异步**：用 `wechatpayv3`，同步调用统一 `asyncio.to_thread` 包裹；客户端懒加载单例，未配置时不构造、不发网络（公钥模式构造也不会下载平台证书）。
- **订单号免存**：`out_trade_no = 订单 UUID 的 hex`（32 位，符合微信 6-32 位与字符集），回调/查单用 `uuid.UUID(hex=...)` 逆向还原，无需额外映射表。
- **回调与查单双保险**：回调（被动）+ 订单状态查询时主动查单（兜底回调丢失）；两条路径都走同一个 `mark_order_paid` 原子流转 + 幂等入账，重复到账不重复发积分。回调对非成功事件/找不到的订单一律回 200，避免微信重试风暴。
- **资金两段式记账**：第三方支付的钱先到微信账户、按周期（约 T+1，扣约 0.6% 手续费）结算到银行。故充值成功记 `借 1002 应收第三方支付 / 贷 2001`；结算到账再 `借 1001 现金 + 借 6603 支付手续费 / 贷 1002`。`1002` 净额 = 微信账户里尚未结算的钱，可与微信「资金账单」对账。**累计充值现金口径改用「已支付充值订单金额之和」**，provider 无关、不受 1001/1002 科目腾挪与结算重复计入干扰。
- **结算对账自动化**：拉微信资金账单 `fundflowbill`（BASIC 账户，`tar_type=None` 拿明文 CSV），按「业务类型」列归类结算/手续费（关键词匹配），按账单日**幂等**入账；账单 CSV 每格有反引号前缀需 `lstrip`，尾部有「资金流水总笔数…」汇总行需按「收支类型」列过滤掉。业务类型中文枚举需以首笔真实账单校验。先「拉取展示」再「确认入账」（同阿里云账单范式），并保留手动结算入口兜底。

## 七、供应商真实余额 / 账单对账（成本闭环）

- **余额对账**：直连云厂商财务 OpenAPI 拉账户余额，与账面预付资产 `1102` 净额对照。
  - 阿里云：BSS `QueryAccountBalance`（系统策略 `AliyunBSSReadOnlyAccess`，或自定义 `bssapi:QueryAccountBalance/QueryBillOverview/...`）。**建独立费用 AccessKey**，与 OSS/短信分开、最小权限。
  - 火山：财务接口 `QueryBalanceAcct`，需账号 IAM AK/SK（**ARK 推理 key 不能查余额**）；SDK 未内置该动作，可在 `BillingService` 上注册自定义 `ApiInfo` 调用。
  - provider 抽象 + TTL 缓存（余额接口有频控、变动不频繁，默认 5 分钟，手动可强刷）；同步 SDK 调用放 `asyncio.to_thread`；未配置凭据直接返回占位、不发请求。
- **基础设施成本计入**：云账户余额下降不只 LLM，还有服务器/存储/带宽。拉月账单（`QueryBillOverview`，成本取 `pretax_amount`），按关键词拆「LLM（已按次计入 COGS，**不重复入账**）vs 基础设施」，把基础设施按账期**幂等**入账为期间费用（借 `6002`/贷 `1102`）。
- **按次估算 vs 账单实际偏差对账**：把账期内**经该厂商平台计费**的模型（按 `base_url` 域名判定，如 `dashscope.aliyuncs.com`＝阿里云百炼，可聚合多家第三方模型）按次估算 COGS 之和，与账单对应金额比，得偏差额/率，用于校准倍率成本基准、识别免费额度。账期按厂商出账时区（如 UTC+8）的自然月匹配。

## 八、易踩的坑

- 服务方法内部 `commit` 会污染测试库：纯逻辑（定价/取整）单独抽函数做无 DB 单测，端到端用「只建 pending 任务、不连 SSE」即可验证前置围栏而不产生 LLM 成本。
- 默认模型可能含高价旗舰：匿名走默认选择时要把可用集合**先限定到允许范围**，否则正常匿名流程会被白名单校验误伤。
- 配置 `lru_cache` 后改文件需重启才生效；做"后台可编辑即时生效"的配置项要在写回文件后显式 `cache_clear()`。
- 路由 router 已带 `prefix` 时，子路由别再重复前缀（否则出现 `/billing/billing/...`）。
- 国内多家模型常**聚合在同一平台计费**（如百炼托管 deepseek/glm/kimi/minimax），账单对账按平台域名归集，而非按模型品牌。
