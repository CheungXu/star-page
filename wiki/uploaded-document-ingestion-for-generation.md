# 上传资料辅助生成页面的处理原则

当用户上传文档辅助生成页面时，推荐把“文件处理”和“页面生成”拆成清晰的两段：先把文件转为可控文本上下文，再把上下文交给页面生成模型。

## 当前策略

- 单次最多允许上传 3 个文件，单文件与单次总大小当前均限制为 50MB。
- Nginx 入口需要设置略高于业务上限的 `client_max_body_size`，当前为 `60m`，避免 multipart 开销导致请求在进入后端前被拦截。
- 原始文件当前不持久保存，只在请求内读取；Office/PDF/HTML 类文件会临时落盘给解析器使用，解析后删除临时文件。
- PDF 当前仅保证可复制文本内容的抽取；扫描版图片 PDF 或加密 PDF 可能解析失败，暂不接入 OCR。
- 新增文件类型通常会引入解析库 extra（例如 PDF 对应 `markitdown[pdf]` 及其 PDF 依赖）。如果生产使用容器部署，除了更新运行中虚拟环境，也必须重新构建并发布后端镜像，否则容器环境仍会缺少解析依赖。
- 后端抽取文本后，如果超过 5000 字符，会先调用 LLM 压缩为“页面生成资料简报”。
- 页面生成模型只接收用户需求和可控长度的资料上下文，不直接吃超长原文。
- 文档压缩属于任务前置阶段，LLM 调用需要重试；除了网络、超时、限流和 5xx 外，也要把“调用成功但正文为空”视为可重试失败。

## 压缩 Prompt 原则

压缩不是通用摘要，而是面向 HTML 页面生成任务的信息重组。应保留：

- 页面主题、受众、业务目标。
- 产品/项目亮点、关键数据、表格结构。
- 文档标题层级、幻灯片页标题、Markdown/HTML 结构。
- 行动号召、品牌语气、视觉和文案约束。
- 缺失信息和不确定点。

应删除：

- 重复寒暄、页眉页脚。
- 低信息密度长段落。
- 与页面生成无关的细节。

## 调试信息

为了排查“模型为什么这样生成”，需要记录：

- 用户原始 prompt。
- 上传文件名。
- 文件抽取文本。
- 触发压缩时的压缩 prompt。
- 最终输入页面生成模型的 prompt。
- 生成 HTML 的 OSS 调试定位信息。
- 模型原始输出文本，用于区分“模型没生成内容”和“HTML 提取/清洗后内容丢失”。

这些信息更适合存入数据库任务记录，不要直接暴露给普通用户。

## 节点可视化

前端建议把整个流程展示为节点流：

```text
上传文件（如有） -> 解析文件 -> 压缩内容 -> 模型思考 -> 模型输出答案 -> 部署
```

其中“模型思考”可以默认展开，展示 reasoning 内容，并提供收起按钮；“部署”可以合并 HTML 上传和数据库记录更新，避免用户看到过多后端实现细节。

## LLM 重试原则

长文压缩和页面生成都依赖 LLM，不能把一次空输出直接当作最终失败：

- 底层 LLM 客户端应统一重试可恢复错误，例如网络失败、超时、协议中断、JSON 解析失败、`429` 和 `5xx`。
- 文档压缩不需要流式展示，建议使用“收集完整正文”的调用方式，并要求正文非空；空正文应重试。
- 页面生成需要流式展示 reasoning 和输出，因此只能在正式 HTML 正文开始前重试；一旦已经开始输出 HTML，再盲目重试容易产生重复或拼接损坏。
- 重试次数、初始退避和最大退避应配置化，避免写死在业务逻辑中。
