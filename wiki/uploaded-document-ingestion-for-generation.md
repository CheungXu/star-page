# 上传资料辅助生成页面的处理原则

当用户上传文档辅助生成页面时，推荐把“文件处理”和“页面生成”拆成清晰的两段：先把文件转为可控文本上下文，再把上下文交给页面生成模型。

## 当前策略

- 单次只允许上传 1 个文件，当前上限为 50MB。
- Nginx 入口需要设置略高于业务上限的 `client_max_body_size`，当前为 `60m`，避免 multipart 开销导致请求在进入后端前被拦截。
- 原始文件当前不持久保存，只在请求内读取；Office 类文件会临时落盘给解析器使用，解析后删除临时文件。
- 后端抽取文本后，如果超过 5000 字符，会先调用 LLM 压缩为“页面生成资料简报”。
- 页面生成模型只接收用户需求和可控长度的资料上下文，不直接吃超长原文。

## 压缩 Prompt 原则

压缩不是通用摘要，而是面向 HTML 页面生成任务的信息重组。应保留：

- 页面主题、受众、业务目标。
- 产品/项目亮点、关键数据、表格结构。
- 文档标题层级、幻灯片页标题、Markdown/HTML 结构。
- 行动号召、品牌语气、视觉和文案约束。
- 缺失信息和不确定点。

应删除：

- 重复寒暄、页眉页脚。
- 低信息密度长段落。
- 与页面生成无关的细节。

## 调试信息

为了排查“模型为什么这样生成”，需要记录：

- 用户原始 prompt。
- 上传文件名。
- 文件抽取文本。
- 触发压缩时的压缩 prompt。
- 最终输入页面生成模型的 prompt。
- 生成 HTML 的 OSS 调试定位信息。

这些信息更适合存入数据库任务记录，不要直接暴露给普通用户。

## 节点可视化

前端建议把整个流程展示为节点流：

```text
上传文件（如有） -> 解析文件 -> 压缩内容 -> 模型思考 -> 模型输出答案 -> 部署
```

其中“模型思考”可以默认展开，展示 reasoning 内容，并提供收起按钮；“部署”可以合并 HTML 上传和数据库记录更新，避免用户看到过多后端实现细节。
