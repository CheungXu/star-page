# 用户维度历史记录持久化原则

## 背景

浏览器 `localStorage` 只能保存当前设备、当前浏览器、当前站点的数据。它适合保存临时 UI 状态，但不适合保存“用户历史记录”这类需要跨设备访问的数据。

在页面生成产品中，历史创建记录应视为用户数据，而不是浏览器缓存。用户换设备、换浏览器或清理站点数据后，仍应能看到自己创建过的页面。

## 推荐边界

适合放在后端数据库：

- 页面历史列表。
- 页面标题、页面状态、访问链接。
- 最近一次生成任务的用户 prompt、上传文件名、任务状态。
- 页面所有者和权限关系。

适合放在浏览器本地：

- 当前设备正在查看的会话 ID。
- 当前设备上的临时 reasoning 展开内容。
- 创建节点的即时 UI 状态。
- 复制成功、输入框内容等短生命周期状态。

## 过渡期默认测试用户

在正式用户系统上线前，可以沿用默认测试用户：

- 后端通过 `ensure_default_user()` 获取当前默认用户。
- 创建页面时把该用户写入 `pages.owner_user_id` 和 `page_permissions`。
- 历史接口按默认用户查询其拥有或有权限访问的页面。

这样可以保持未来用户体系的表结构和权限边界，后续接入真实登录用户时，只需要替换“当前用户来源”，而不是重写页面历史模型。

## 接口设计

历史列表建议返回前端恢复历史会话所需的最小数据：

- `page_id`
- `task_id`
- `title`
- `prompt`
- `file_names`
- `page_url`
- `page_status`
- `generation_status`
- `created_at`
- `updated_at`

前端点击历史时，优先使用本机缓存的会话细节；如果当前设备没有缓存，则用后端返回的数据重建一个可预览的会话。

## 经验总结

判断一个状态放哪里，可以用一句话：

如果用户换设备后仍然期望看到它，就应该放数据库；如果只影响当前设备的即时交互体验，才适合放 `localStorage`。
