# 用户维度历史记录持久化原则

## 背景

浏览器 `localStorage` 只能保存当前设备、当前浏览器、当前站点的数据。它适合保存临时 UI 状态，但不适合保存“用户历史记录”这类需要跨设备访问的数据。

在页面生成产品中，历史创建记录应视为用户数据，而不是浏览器缓存。用户换设备、换浏览器或清理站点数据后，仍应能看到自己创建过的页面。

## 推荐边界

适合放在后端数据库：

- 会话历史列表，以及收藏、软删除等跨设备一致的历史管理状态。
- 会话标题、最近更新时间、最近一轮模型选择和生成状态。
- 会话内批次、页面节点、生成任务、页面访问链接。
- 页面所有者和权限关系。

适合放在浏览器本地：

- 当前设备正在查看的会话 ID。
- 当前设备上的临时 reasoning 展开内容。
- 创建节点的即时 UI 状态。
- 复制成功、输入框内容等短生命周期状态。

## 过渡期默认测试用户

在正式用户系统上线前，可以沿用默认测试用户：

- 后端通过 `ensure_default_user()` 获取当前默认用户。
- 创建页面时把该用户写入 `conversations.owner_user_id`、`pages.owner_user_id` 和 `page_permissions`。
- 历史接口按默认用户查询其拥有的会话，页面访问仍按页面权限控制。

这样可以保持未来用户体系的表结构和权限边界，后续接入真实登录用户时，只需要替换“当前用户来源”，而不是重写页面历史模型。

## 接口设计

历史列表建议返回前端展示和恢复历史会话所需的最小数据：

- `conversation_id`
- `title`
- `is_favorite`
- `model_keys`
- `node_count`
- `latest_batch_status`
- `created_at`
- `updated_at`

前端点击历史时，应通过会话详情接口读取批次、页面节点和生成任务信息，再重建一个可预览的会话。浏览器本地缓存只用于恢复当前设备正在查看的即时 UI 状态。

## 历史管理状态

删除历史记录优先使用软删除字段，例如 `deleted_at`，让记录默认不再出现在历史列表中，同时保留页面产物、排查线索和未来恢复空间。

收藏属于用户对历史记录的管理偏好，也应持久化到数据库，例如会话级 `is_favorite`。收藏筛选和关键词检索应下推到后端查询参数，避免只在前端过滤当前页的前几十条结果。

## 经验总结

判断一个状态放哪里，可以用一句话：

如果用户换设备后仍然期望看到它，就应该放数据库；如果只影响当前设备的即时交互体验，才适合放 `localStorage`。
