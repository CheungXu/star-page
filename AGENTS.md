# 目录说明


| 目录        | 说明                                                                                                |
| --------- | ------------------------------------------------------------------------------------------------- |
| `config/` | 配置文件目录，保存各类配置信息                                                                                   |
| `script/` | 脚本目录，存放临时支持任务的脚本                                                                                  |
| `code/`   | 代码目录，存放主框架模块                                                                                      |
| `rules/`  | 项目开发准则和规范                                                                                         |
| `data/`   | 数据目录，存放相关数据                                                                                       |
| `doc/`    | 日常工作文档，包含分析报告、工作记录等，更偏工作细节，目录结构 `doc/YYYYMMDD/`                                                   |
| `wiki/`   | 团队 wiki 知识库；区别于日常流水账，记录最有价值、最精华、最能帮助他人的知识。本仓库通用知识写在 `wiki/` 下，各子项目专有知识写在 `wiki/{project_name}/` 下 |
| `skills/` | 封装好的技能，每个技能一个文件夹，例如`skills/add_xxx/`                                                              |
| `images/` | 图片资源信息                                                              |

# 工作原则

1. **工作语言为中文。**
2. **任务执行要求**：
  - 按理解和规划合理管理各目录下的文件增删改，目录请严格按以上目录维护，工作中只能在以上目录中新增修改或删除文件，不在主目录下进行新增文件/目录或处理。如需新增目录，先进行询问，确认后新增。
  - 每次任务按理解和规划，合理管理各目录下的文件增删改，保持良好的项目架构。
  - 为避免过多信息堆积到最上层，代码下每一级目录可以有一个readme，但要按照你的理解合理规划，平衡单文件过于聚集和多文件信息过于分散。
3. **工作知识沉淀**：
  - 不同文档的区别：
    - `code/`、`script/` 等目录内的 readme 或说明：主要**面向开发者**，提供使用方法、代码/模块架构、配置信息等开发细节，便于高效开发与维护。
    - `doc/{YYYYMMDD}/xxx.md`：主要**面向项目管理**，隐去具体技术与代码细节，提供项目背景、方案架构、实施规划与流程、进度等，便于非技术管理者了解状态并参与规划与决策。
    - `wiki/`：知识库；readme 与 doc 主要服务本项目迭代，wiki 则抽象**可跨项目复用**的知识，帮助其他任务快速了解通用经验、复用做法，提升整体效率。
  - 每次完成工作后，思考：
    - 代码或模块变更后，是否需更新对应 readme。
    - 是否有信息需记入 `doc/`，以便后续继续该任务时有上下文。
    - 是否有核心知识需记入 `wiki/`，以便其他任务或项目复用。
  - 当我主动说“沉淀”、“沉淀一下”时，指的是：回顾上一次沉淀到当前的所有对话，将相关信息/经验/知识/技能，按需更新到对应的readme、skill、doc和wiki中。
